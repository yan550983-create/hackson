# AGENTS.md

## 项目目标

本项目是 `event-networking-assistant`，中文名为“活动情报与低能耗社交助理”。目标是帮助用户在线下活动前后完成信息判断、交流取舍和关系沉淀。它不是普通社交软件，也不是泛泛的人脉 CRM，而是活动情报系统、低能耗社交决策系统和会后复盘系统。

## 用户使用场景

用户经常参加 AI、出海、TikTok Shop、跨境电商、创业、投资、黑客松、青年科创、行业分享类活动。输出要帮助用户判断：谁值得聊，为什么值得聊，怎么聊，聊完以后怎么跟进。

## 输出语言要求

- 用户可见内容必须用中文。
- 代码、字段名、目录名、文件名必须用英文。
- 报告、网页文案、模板说明要优先服务“低能耗但高信息判断力”的使用者。

## 文件命名规范

- 目录名、脚本名、字段名使用英文小写和下划线或连字符。
- 输出报告文件名包含日期、活动名和类型，例如 `2026-06-01-ai-event-pre-brief.md`。
- 私人输出默认写入 `outputs/`，并通过 `.gitignore` 避免提交 Markdown、TXT、JSON 报告。

## 隐私和安全边界

- 只使用公开信息和用户明确提供的材料。
- 不推断私人隐私，不做人脸识别，不做声纹识别。
- 不偷拍、偷录、监听，不做违反网站规则的爬虫。
- 不从录音或转写中推断敏感个人信息。
- 不把普通寒暄过度解读成人脉价值。
- 录音/转写功能默认关闭，只处理用户主动提供的内容。
- 不自动发送邮件、微信或任何跟进消息。
- 不默认上传任何用户数据。

## 报告输出标准

- 所有事实、推测、待确认信息必须分开。
- 所有事实性信息尽量附来源；没有来源就标注“来源待补充”。
- 所有“可能到场人员”都必须标注为推测，并写出依据。
- 活动分析要把活动看作“信息场”和“关系场”，不是普通社交场。
- 如果信息不足，列出待补充信息，不要编造。

## 不允许过度推测

- 不要默认所有人都值得聊。
- 不要把公开身份等同于真实合作意愿。
- 不要把礼貌交换联系方式等同于高价值关系。
- 对推测必须给出依据和置信度。

## 不允许泛泛社交建议

- 不要写“多认识朋友”“扩大人脉”这类泛泛建议。
- 不要鼓励无目标地交换名片或加微信。
- 要帮助用户节省精力，做筛选和取舍。

## 代码原则

- 代码尽量简单、可读、本地优先。
- 第一版不引入后端、数据库、登录、真实爬虫、真实音频转写或 API key。
- 网页 demo 使用原生 HTML/CSS/JavaScript。
- Python 脚本不联网，缺字段时输出“待补充”而不是中断。
